Language

Avoid mixing pronouns in the same line of text. An example of pronoun mixing would be "Change your preferences using My Account."

Avoid the pronoun "we," instead maintain focus on the user and what they can do with the application. The only exception to this rule is if there is a response from the organization required. An example would be, the user submits a question through email or FAQ and an automatic response generates, "We have received your inquiry and will respond as soon as possible." In this instance, the use of "we" is appropriate.

Be concise. Write in small, scannable phrases to ease navigation throughout the application. Limit concepts introduced in a single phrase. Eliminate common introductory phrases and keep language direct. (i.e. "Save changes?" Rather than "Would you like to save your changes?")

Keep sentences in the present tense as much as possible. If future or past tenses are needed, stick with simple verb forms.

Use language that is suitable for all reading levels. Text should be comprehensible to novice and advanced English speakers. Avoid the use of jargon as it limits audience comprehension.

Use consistent verbs across the application to reduce confusion.

When necessary, use numerals instead of the words for numbers. The only exception to this would be if directions are telling the user to "enter three 2's" or something similar.

When phrases include directions, begin with the objective and end with the instruction. An example would be "To search a case by date filed, select the date from the calendar" rather than "Select the date from the calendar to search a case by date filed."

Avoid the use of absolutes such as "never" or "always."

Avoid the use of codes for simple information that may be pulled from Contexte data. Codes such as "F" or "M" for gender may be pulled, but to enhance user clarity those should reflect "Female" or "Male."

Tone

The tone of the application should match its design. Keep it straight-forward, helpful, respectful, and trustworthy.

Be humble. Tell the user what a feature does without bragging or over-sensationalizing.

To reassure the user, keep the tone positive. For example, "try again" rather than "action failed."

Communicate essential details. Omit extraneous information so the user can concentrate on the task at hand.

Capitalization and punctuation

Use sentence-style capitalization. In headings, titles, labels, and menu items capitalize the first letter of the first word and then use lowercase thereafter. Avoid the use of all capitalizations except where specified in the style guide, i.e. buttons.

To help users scan information at a glance, skip unnecessary punctuation.

Avoid using periods in solitary sentences in these elements:

  • Labels
  • Hover text
  • Bulleted lists
  • Dialog body text

Periods should be used on:

  • Multiple sentences
  • Any sentence followed by a link

For ease of comprehension, use contractions. There may be instances when you want to emphasize information like "do not delete," in which case "do not" is a better use than a contraction (don't).

Avoid exclamation points except in greetings (i.e. "Welcome!").
